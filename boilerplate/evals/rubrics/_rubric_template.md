## 평가 차원

### 1. 완전성 (Completeness) - 가중치 40%

**평가 기준**: 요구사항을 얼마나 완전하게 충족했는가?

| 점수 | 기준 |
|------|------|
| 5 (Excellent) | 모든 요구사항 충족, 엣지 케이스까지 처리 |
| 4 (Good) | 핵심 요구사항 모두 충족, 일부 세부사항 누락 |
| 3 (Adequate) | 대부분의 요구사항 충족, 몇 가지 누락 |
| 2 (Poor) | 일부 요구사항만 충족, 중요한 부분 누락 |
| 1 (Fail) | 요구사항 거의 충족하지 못함 |

**체크리스트**:
- [ ] {{requirement-1}}
- [ ] {{requirement-2}}
- [ ] {{requirement-3}}
- [ ] 엣지 케이스 처리

---

### 2. 코드 품질 (Code Quality) - 가중치 30%

**평가 기준**: 코드가 얼마나 깔끔하고 유지보수 가능한가?

| 점수 | 기준 |
|------|------|
| 5 (Excellent) | 모범적인 코드, 적절한 추상화, 명확한 구조 |
| 4 (Good) | 깔끔한 코드, 일관된 스타일, 좋은 네이밍 |
| 3 (Adequate) | 동작하는 코드, 일부 개선 여지 있음 |
| 2 (Poor) | 동작하지만 가독성/구조 문제 |
| 1 (Fail) | 이해하기 어렵거나 심각한 품질 문제 |

**체크리스트**:
- [ ] 일관된 코딩 스타일
- [ ] 명확한 변수/함수 네이밍
- [ ] 적절한 모듈화/추상화
- [ ] 불필요한 복잡성 없음
- [ ] 에러 처리 적절함

---

### 3. 정확성 (Correctness) - 가중치 30%

**평가 기준**: 결과물이 얼마나 정확한가?

| 점수 | 기준 |
|------|------|
| 5 (Excellent) | 완벽하게 정확, 모든 케이스에서 올바른 결과 |
| 4 (Good) | 대부분 정확, 사소한 이슈만 있음 |
| 3 (Adequate) | 기본적으로 정확, 일부 케이스에서 오류 |
| 2 (Poor) | 여러 오류 존재, 일부만 정확 |
| 1 (Fail) | 대부분 부정확하거나 심각한 오류 |

**체크리스트**:
- [ ] 기본 기능 정상 동작
- [ ] 테스트 통과
- [ ] 엣지 케이스에서도 정확
- [ ] 예상치 못한 입력 처리

---

## 최종 점수 계산

```
최종 점수 = (완전성 × 0.4) + (코드 품질 × 0.3) + (정확성 × 0.3)
```

## 통과 기준

- **Pass**: 최종 점수 ≥ 3.5
- **Conditional Pass**: 최종 점수 3.0-3.4 (검토 필요)
- **Fail**: 최종 점수 < 3.0

---

## 평가 시 주의사항

1. **관대함의 원칙**: 애매한 경우 에이전트에게 유리하게 해석
2. **맥락 고려**: Task의 난이도와 제약조건 고려
3. **일관성**: 동일한 기준을 모든 결과물에 적용
4. **근거 제시**: 각 점수에 대한 구체적인 근거 기록
